## 背景需求

当前 Agent 记忆系统面临的核心问题是**缺乏统一的操作标准**。不同框架各自实现记忆的增删改查，导致：

1. **互操作性差** — 难以在不同记忆系统间迁移
2. **开发成本高** — 每套系统需要重新设计 API
3. **验证困难** — 缺乏统一的契约来验证记忆操作的有效性

## 目标

Text2Mem 旨在为所有记忆系统定义一套**标准化的"操作指令集"**，通过 12 个原子操作 + 五元 JSON 契约 + 双层验证，实现：

- 统一的记忆操作接口
- 可验证的记忆存储
- 跨框架的互操作性

## 设计方案

### 核心架构

**1. 12 个原子操作**

Text2Mem 定义了 12 个记忆操作的原子指令，覆盖记忆全生命周期：

| 操作类型 | 原子操作 |
|----------|----------|
| 写入 | `STORE`, `APPEND`, `UPDATE`, `DELETE` |
| 读取 | `RETRIEVE`, `SEARCH`, `FILTER` |
| 管理 | `ARCHIVE`, `RESTORE`, `MERGE`, `SPLIT`, `VALIDATE` |

**2. 五元 JSON 契约**

每个记忆操作通过统一的五元 JSON 结构定义：

```json
{
  "op": "STORE",
  "entity": "user_preference",
  "content": "Prefers dark mode",
  "metadata": {
    "timestamp": "2026-04-15T12:00:00Z",
    "source": "explicit_user_input"
  },
  "constraints": {
    "ttl": "30d",
    "priority": "high"
  }
}
```

五元组解析：
- **op** — 操作类型
- **entity** — 记忆实体类型
- **content** — 记忆内容
- **metadata** — 元数据（时间戳、来源等）
- **constraints** — 约束条件（TTL、优先级等）

**3. 双层验证**

- **语法验证** — JSON Schema 验证契约格式
- **语义验证** — LLM 判断操作是否符合业务逻辑

```bash
git clone https://github.com/text2mem/text2mem.git
cd text2mem
pip install -e .
```

**基础使用示例：**

```python
from text2mem import MemoryOperator

# 初始化操作器
operator = MemoryOperator()

# 存储记忆
result = operator.execute({
    "op": "STORE",
    "entity": "user_preference",
    "content": "User prefers morning meetings",
    "metadata": {"source": "conversation"},
    "constraints": {}
})

# 检索记忆
results = operator.execute({
    "op": "RETRIEVE",
    "entity": "user_preference",
    "content": "meeting preferences",
    "metadata": {},
    "constraints": {"limit": 5}
})
```

## 优缺点分析

### 优点

1. **标准化程度高** — 12 个原子操作覆盖记忆全生命周期
2. **可验证性强** — 双层验证确保数据质量
3. **互操作性好** — 统一契约便于跨框架集成
4. **灵活性好** — 支持自定义约束条件

### 缺点

1. **学习成本** — 需要理解 12 个原子操作和契约格式
2. **实现复杂度** — 双层验证增加了系统复杂度
3. **生态不成熟** — 作为新项目，集成案例较少
4. **性能开销** — 验证层带来额外计算开销

## 平替对比

| 框架 | 标准化程度 | 验证机制 | 复杂度 | 适用场景 |
|------|-----------|----------|--------|----------|
| Text2Mem | 高（12 原子操作） | 双层验证 | 中高 | 需要标准化的企业级应用 |
| Mem0 | 中（工厂模式） | 单一验证 | 中 | 快速原型开发 |
| Letta | 低（OS 风格） | 无 | 高 | 需要深度个性化的场景 |
| ReMe | 低（文件风格） | 无 | 低 | 需要用户透明性的场景 |
| memU | 低（Agent 风格） | 无 | 高 | 需要主动学习的场景 |

## 使用场景

- **企业级 Agent** — 需要跨系统统一记忆管理
- **多 Agent 协作** — 需要标准化的记忆交互协议
- **记忆系统迁移** — 需要将数据从一套系统迁移到另一套
- **合规要求高** — 需要可验证的记忆操作审计

## 成本评估

- **开源免费**：Apache 2.0 许可证
- **运行成本**：主要是 LLM API 调用费用（用于语义验证）
- **自托管成本**：需要向量数据库（Milvus/Qdrant）+ LLM API

## 学习曲线

- **入门难度**：中等 — 需要理解 12 个原子操作和 JSON 契约
- **进阶功能**：高 — 需要掌握双层验证机制
- **推荐学习路径**：先了解 Mem0 基础，再学习 Text2Mem 标准化设计
